# 上传与大文件

AsterDrive 不要求你手动选择上传方式。
你只需要上传文件，系统会根据文件大小和当前存储策略自动选择合适的处理方式。

## 普通用户平时只需要知道这些

- 小文件通常会很快完成
- 大文件通常会自动分片上传
- 上传中断后，系统会尽量提示继续未完成的上传
- 继续上传时，重新选择同一个文件即可

日常使用时，你不需要区分“普通上传”“分片上传”或“S3 直传”。

## 大文件上传为什么会占用本地磁盘

即使文件最终要放到 S3 / MinIO，服务端在某些上传策略下仍然会使用本地临时目录。

最常见的情况有:

- 使用默认本地存储
- 使用 S3 的 `proxy_tempfile`
- 经过服务端的分片上传

所以管理员要提前确认服务器本地磁盘还有足够空间，尤其是:

- `data/.tmp`
- `data/.uploads`

## 如果你使用 S3 / MinIO，需要先决定上传策略

管理员在存储策略里最常见的三种 S3 上传方式是:

### 服务端暂存上传

浏览器先把文件传给 AsterDrive，服务端再写入 S3。兼容性最好，但会占用本地临时磁盘。

### 服务端流式中继

浏览器先把文件传给 AsterDrive，服务端直接中继到 S3，不落本地临时分片目录。

### Presigned 直传

浏览器直接把文件或分片上传到 S3 / MinIO。
这种方式通常最省服务端带宽，但对象存储必须先配好浏览器上传需要的 CORS。

## 上传失败时，先按这个顺序检查

1. 当前使用的是哪条存储策略
2. 这条策略的单文件大小上限是否足够
3. 反向代理的请求体大小和超时时间是否足够
4. 如果走的是 S3 Presigned，对象存储的 CORS 是否已经配置正确
5. 用户配额是否已经达到上限

## 什么时候应该改配置

如果用户经常上传大文件，管理员通常只需要关注这几处:

- `管理 -> 存储策略` 里的单文件大小上限和分片大小
- 服务器本地临时目录的可用空间
- 反向代理的上传大小和超时
- S3 / MinIO 的 CORS
